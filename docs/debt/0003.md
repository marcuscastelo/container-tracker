# Dívidas Técnicas — Container Tracker Agent (MVP One-Click)

---

# 🔴 1. Segurança

## 1.1 Token embutido no installer (INSTALLER_TOKEN)
**Estado atual (MVP):**
- Token de bootstrap em texto plano no bundle.
- Fica presente no `bootstrap.env` até enrolment.

**Risco:**
- Extração do token do .exe ou do `ProgramData`.
- Uso indevido se vazado.

**Mitigação MVP:**
- Token escopado por tenant.
- Token revogável.
- (Ideal) token de curta duração.
- Apagar ou invalidar `bootstrap.env` após enrolment.

**Próximo passo pós-MVP:**
- Enrolment via OAuth/device code.
- Download autenticado no SaaS.
- Code signing obrigatório.

---

## 1.2 Sem code signing
**Estado atual:**
- Setup.exe e binários não assinados.

**Risco:**
- SmartScreen warning.
- Bloqueio por AV corporativo.
- Desconfiança do cliente.

**Próximo passo:**
- Assinatura Authenticode com timestamp.

---

## 1.3 Secrets em texto plano em `ProgramData`
`config.env` conterá `AGENT_TOKEN`.

**Risco:**
- Usuário local com privilégios pode ler token.
- Exfiltração lateral.

**Mitigação MVP:**
- ACL restritiva: SYSTEM + Administrators only.

**Próximo passo:**
- DPAPI ou Windows Credential Manager.
- Encrypt-at-rest.

---

## 1.4 Logs potencialmente expondo secrets
Se algum erro logar headers ou env completo.

**Risco:**
- Vazamento via logs.

**Mitigação obrigatória já:**
- Nunca logar tokens.
- Sanitização de erros HTTP.

---

# 🟠 2. Operacional

## 2.1 Updater é stub
**Estado atual:**
- Apenas loga “NO UPDATES”.
- Não baixa nada.
- Não reinicia serviço.

**Risco:**
- Drift de versão manual.
- Atualizações dependem de novo installer.

**Próximo passo:**
- Manifest versionado.
- Download seguro.
- Atomic swap.
- Restart controlado.

---

## 2.2 Sem rollback de versão
Se update futuro quebrar, não há rollback.

**Risco:**
- Agente pode ficar indisponível.

**Próximo passo:**
- Estrutura de versão A/B.
- Retenção de versão anterior.

---

## 2.3 Enrolment depende de backend idempotente
Se o endpoint não for idempotente:

**Risco:**
- Múltiplos agents criados para mesma máquina.

**Próximo passo:**
- Fingerprint + idempotência server-side.

---

## 2.4 Retry sem política formalizada
Falamos em backoff exponencial, mas ainda não está especificado.

**Risco:**
- Spam no backend.
- DOS acidental.

**Próximo passo:**
- Definir estratégia exata (cap + jitter).

---

## 2.5 Sem heartbeat / health monitor
Hoje o backend só recebe snapshots.

**Risco:**
- Agent offline não detectado.
- Falha silenciosa.

**Próximo passo:**
- Endpoint `/api/agent/heartbeat`
- Alerta de agent offline.

---

# 🟡 3. Produto / UX

## 3.1 Zero UI de diagnóstico
Usuário não tem painel local.

**Risco:**
- Dificuldade de troubleshooting.
- Suporte dependente de acesso remoto.

**Próximo passo:**
- Mini UI local de status (read-only).
- Ou comando CLI `agent --status`.

---

## 3.2 Installer por tenant (escala futura)
Hoje 1 cliente → OK.  
Futuro multi-tenant → precisa pipeline.

**Risco:**
- Processo manual vira gargalo.
- Erro humano ao gerar installer.

**Próximo passo:**
- Pipeline automatizado por tenant.
- Download autenticado no SaaS.

---

# 🟢 4. Arquitetura / Coerência com Domínio

## 4.1 Bootstrap vs Config Merge
Hoje:
- Se `config.env` existe, não re-enrola.
- Sem schema versioning.

**Risco:**
- Mudança futura de config quebra agentes antigos.

**Próximo passo:**
- Versionamento de schema no config.
- Migration controlada.

---

## 4.2 Sem materialização de versão ativa do agent
Agent não informa versão estruturada ao backend.

**Risco:**
- Difícil saber quem está desatualizado.

**Próximo passo:**
- Enviar `agent_version` no enrolment e heartbeat.

---

# 🔵 5. Infra Windows

## 5.1 ACL não formalizada ainda
Falamos em restringir ProgramData, mas não definimos no `installer.iss`.

**Risco:**
- Permissões amplas por padrão.

**Próximo passo:**
- Configurar ACL no installer via `[Code]`.

---

## 5.2 Detecção Chrome para MAERSK heurística
Se `MAERSK_ENABLED=1`:

**Risco:**
- Falso negativo em ambientes corporativos.
- Caminhos customizados.

**Próximo passo:**
- Documentar dependência.
- Tornar validação mais robusta.

---

# 🟣 6. Decisões Estratégicas Conscientemente Adiadas

1. Sem multi-channel release.
2. Sem política de retry da task do updater.
3. Sem assinatura de manifesto de update.
4. Sem criptografia adicional além de TLS padrão.
5. Sem UI de configuração remota.
6. Sem observabilidade estruturada (OpenTelemetry, etc.).

---

# 📦 Resumo Executivo

## Críticas (resolver antes de escalar)
- Token embutido.
- Code signing.
- ACL restritiva.
- Idempotência do enrolment.
- Backoff formalizado.

## Importantes (pós-MVP imediato)
- Heartbeat.
- Version tracking.
- Update real com manifest.
- Encrypt-at-rest.

## Evolução futura
- Pipeline automatizado por tenant.
- UI local de diagnóstico.
- Release channel estruturado.
- Observabilidade avançada.

---

**Conclusão:**  
Para 1 cliente no MVP, o risco é controlado e consciente.  
As dívidas são majoritariamente de segurança operacional e ciclo de update —  
não comprometem domínio nem arquitetura do Container Tracker.
