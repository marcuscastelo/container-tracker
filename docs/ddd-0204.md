# DDD — Bounded Contexts do Container Tracker

Este documento mapeia **todos os bounded contexts** do projeto, com foco em clareza de domínio, responsabilidades, contratos e integrações. Ele é pensado para **onboarding de devs** e **assistentes LLM**, e serve como referência viva para decisões arquiteturais.

> Princípios:
>
> * Um contexto = um **modelo mental consistente**.
> * Comunicação entre contextos **sempre explícita** (eventos/ACLs).
> * Nenhum contexto “vaza” regras para outro.

---

## 1) Process Management

**Responsabilidade:** Existência e lifecycle do *Shipment* (processo).

**Entidades-chave:**

* Shipment (Process)

**Invariantes:**

* Shipment **não tem status próprio**.
* Shipment pode existir incompleto.

**Comandos típicos:**

* CreateShipment
* UpdateShipmentMeta (origem/destino/carrier)

**Eventos emitidos:**

* ShipmentCreated
* ShipmentUpdated

**Integrações:**

* → Container Management
* → Alerting (indireto, via containers)

---

## 2) Container Management

**Responsabilidade:** Unidade operacional real (container).

**Entidades-chave:**

* Container

**Invariantes:**

* Container é globalmente único.
* Container **sempre pertence a um Shipment**.

**Comandos típicos:**

* AddContainerToShipment
* RemoveContainerFromShipment

**Eventos emitidos:**

* ContainerAdded
* ContainerRemoved

**Integrações:**

* ← Process Management
* → Event Timeline
* → Status Derivation

---

## 3) Event Timeline (Canonical Events)

**Responsabilidade:** Verdade histórica imutável.

**Entidades-chave:**

* Event (ACTUAL | EXPECTED)

**Invariantes:**

* Eventos são **imutáveis**.
* Ordenação sempre por `event_time`.
* Eventos de sistema **não afetam status**.

**Fontes:**

* Manual
* Carrier APIs
* Sistema

**Integrações:**

* ← Carrier Integration (via ACL)
* → Status Derivation
* → Alerting

---

## 4) Carrier Integration

**Responsabilidade:** Conversar com o mundo externo (armadores).

**Subcontextos (por carrier):**

* Maersk Adapter
* MSC Adapter
* CMA CGM Adapter

**Padrão:**

* Anti-Corruption Layer (ACL)
* Normalização → Canonical Event

**Invariantes:**

* Payload bruto sempre preservado.
* Nenhuma regra de domínio aqui.

**Integrações:**

* → Event Timeline

---

## 5) Status Derivation Engine

**Responsabilidade:** Derivar estado atual a partir de eventos.

**Entidades-chave:**

* DerivedContainerState

**Regras:**

* Função pura: `deriveContainerState(events[])`
* Usa **apenas ACTUAL logísticos**.

**Outputs:**

* status
* eta_final
* flags (missing_eta, stale_data)

**Integrações:**

* ← Event Timeline
* → Alerting
* → Read Models (UI)

---

## 6) Alerting

**Responsabilidade:** Priorizar atenção humana.

**Entidades-chave:**

* Alert

**Invariantes:**

* Alertas são **deriváveis novamente**.
* Todo alerta aponta para eventos concretos.

**Categorias:**

* eta
* movement
* customs
* data
* status

**Lifecycle:**

* create → acknowledge → expire

**Integrações:**

* ← Event Timeline
* ← Status Derivation
* → Notification

---

## 7) Notification

**Responsabilidade:** Entregar alertas fora do sistema.

**Canais (v1):**

* Email

**Invariantes:**

* Não cria alertas.
* Não contém regra de severidade.

**Integrações:**

* ← Alerting

---

## 8) Read Models / Dashboard

**Responsabilidade:** Projeções para leitura (UI).

**Entidades-chave:**

* ProcessSummary
* ContainerRow
* AlertBadge

**Invariantes:**

* Somente leitura.
* Pode ser reconstruído.

**Integrações:**

* ← Process Management
* ← Container Management
* ← Status Derivation
* ← Alerting

---

## 9) Search & Indexing

**Responsabilidade:** Busca rápida e filtros.

**Índices:**

* container_number
* shipment_id
* carrier
* status

**Integrações:**

* ← Read Models

---

## 10) Data Quality & Observability (futuro)

**Responsabilidade:** Confiança do dado.

**Sinais:**

* carrier silent
* dados inconsistentes

**Integrações:**

* ← Event Timeline
* → Alerting

---

## 11) Audit & Compliance (futuro)

**Responsabilidade:** Rastreabilidade total.

**Entidades:**

* AuditLog

**Integrações:**

* ← Todos os contextos que mutam estado

---

## Mapa de Relacionamentos (alto nível)

```
Carrier APIs
    ↓
Carrier Integration (ACL)
    ↓
Event Timeline ──→ Status Derivation ──→ Alerting ──→ Notification
        ↓                     ↓
   Read Models / Dashboard  Search
```

---

## O que NÃO é bounded context (decisão consciente)

* UI (é consumidor de read models)
* Auth / Billing (fora do escopo atual)
* BI / Analytics pesado

---

## Regra de Ouro para Evolução

> Se uma regra começa a “incomodar” mais de um contexto,
> ela provavelmente está no lugar errado.

Esse mapa deve ser revisitado a cada nova fase do roadmap.
